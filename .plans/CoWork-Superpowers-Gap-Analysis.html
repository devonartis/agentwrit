Superpowers + Context-First Development

Gap analysis of the Superpowers skill framework against the full development lifecycle — and the 5 new skills needed to close the gaps.
AgentAuth Project · March 2026 · Based on obra/superpowers + real session experience

What Superpowers Already Covers (and What It Doesn't)

Superpowers is an excellent skill framework. It covers the design → plan → build → verify pipeline thoroughly, with strong opinions on TDD, verification evidence, code review, and subagent orchestration. But it focuses on the execution side — how to build correctly.

What's missing is the discovery and impact analysis side — how to understand what exists before you build. This is the gap we hit repeatedly during the AgentAuth session. Every mid-session correction ("you missed the full flow," "we don't use SPIRE," "we can't fix one thing and break other things") traces back to insufficient pre-build context.

The analogy: Superpowers teaches the contractor how to build (measure twice, cut once, test-drive every board). But it doesn't include a site survey skill — the step where you walk the property and map what's already there before you start cutting. That's the gap.

The 5 New Skills Needed

These fill the gaps in Superpowers. They follow the same conventions: announcement at start, specific deliverables, clear handoff to the next skill, and red flags section.

1. codebase-archaeology
Phase 1: DISCOVER · Step 1.1 + 1.3
Maps the territory before any design begins. The agent explores the codebase and produces a comprehensive inventory of every capability, endpoint, flow, data model, security property, and integration point. Then compares it against any source pattern or standard to produce a compliance matrix.

Why it's missing: Superpowers' brainstorming skill explores project context, but it's focused on understanding the idea, not inventorying what exists. Those are different activities. Brainstorming asks "what should we build?" This skill answers "what's already here?"

Real example: If we'd had this skill before the AgentAuth session, we would have mapped all 17 endpoints, 13 capabilities, and 10 security invariants BEFORE designing anything. Every mid-session correction ("you missed the full flow," "we don't use SPIRE") would have been caught upfront.
Deliverables: capability-inventory.md, compliance-matrix.md

Handoff: brainstorming skill (feeds the design phase with complete context)
2. impact-analysis
Phase 2: DESIGN · Step 2.2
Proves the proposed design doesn't break existing capabilities. Takes the capability inventory (from codebase-archaeology) and the design (from brainstorming) and produces a capability-by-capability matrix: PRESERVED / ENHANCED / NEW / REMOVED. Also verifies security invariants.

Why it's missing: Superpowers has strong verification for new code (TDD, verification-before-completion). But nothing for existing code — proving what's already working will continue to work after the change.

Real example: "We can't fix one thing and break other things." The Full Pattern Flow section — 17-row capability table + 10 security invariants — is exactly what this skill produces.

The rule: If you can't fill the preservation matrix completely, you don't understand the change well enough to build it.
Deliverables: impact-analysis.md, security-invariants.md

Handoff: writing-plans skill (feeds task breakdown with change/no-change lists)
3. design-review
Phase 2: DESIGN · Step 2.4
Architecture-level review of the design BEFORE coding starts. Different from requesting-code-review (which reviews CODE after it's written). This reviews the DESIGN — the PRD, impact analysis, and implementation plan — looking for gaps, contradictions, and missing edge cases.

Why it's missing: Superpowers' requesting-code-review dispatches a code-reviewer subagent to look at diffs. This skill dispatches a design-reviewer subagent to look at the plan documents.

Real example: The 3rd-party developer found 7 implementation gaps AFTER the design was "done." If we'd had a design-review gate, those gaps would have been caught before task breakdown.

Format: Reviewer identifies gaps → each gap gets a decision → decisions documented with rationale and phase assignment.
Deliverables: review-decisions.md, updated design docs

Handoff: writing-plans skill (design is now review-approved, ready for task breakdown)
4. task-briefing
Phase 3: BUILD · Step 3.0
Compiles a single TASK-BRIEF.md from all discovery and design outputs. This is the bridge between human-led design and agent-led implementation. The brief tells the agent: here's what exists, here's what we're building, here's what you must not touch, and here's how to verify.

Why it's missing: writing-plans creates detailed implementation plans. But the plan assumes the agent knows the codebase. TASK-BRIEF.md bridges the gap — it's "CLAUDE.md for this specific feature."

Key sections: Context + Current State links + Design Decisions + DO NOT TOUCH list + CHANGE list + NEW list + Verification commands + Success criteria.

Cross-agent compatible: Any agent that reads files can follow TASK-BRIEF.md. Claude reads it via CLAUDE.md reference, Cursor via .cursorrules, etc.
Deliverables: TASK-BRIEF.md

Handoff: executing-plans or subagent-driven-development (agent reads brief, then executes plan)
5. regression-verification
Phase 4: VERIFY · Step 4.2 + 4.3
Proves existing capabilities still work after the change. Takes the impact analysis matrix and runs targeted tests for every capability marked PRESERVED or ENHANCED. Also compares the actual implementation against the design to detect drift.

Why it's missing: verification-before-completion answers "does the new thing work?" This skill answers "does the old thing still work?" and "did we build what we designed?"

Real example: After building the app registration feature, we'd run regression-verification to prove Ed25519 challenge-response, scope enforcement, delegation chains, revocation, and audit trail all still work exactly as before.

Also includes: Design-vs-actual comparison — diff the implementation against the implementation details doc. Flag any "do not touch" files that were modified, any unplanned changes, any design drift.
Deliverables: regression-results.md, post-build-review.md

Handoff: finishing-a-development-branch (all verification complete, ready to merge)

Enhancements to Existing Superpowers Skills

Besides the 5 new skills, some existing Superpowers skills could use targeted enhancements to better support the full lifecycle:

Existing Skill Enhancement Why
brainstorming Add "What MUST NOT Change" section to design doc template. Add ADR (Architecture Decision Record) format for "do we need X?" questions. Constraints and opt-outs need a formal home. The SPIRE confusion and sidecar question were both ADRs without a format.
writing-plans Add "DO NOT TOUCH" file list per task. Add "files NOT affected" alongside "files to change." Agents sometimes refactor files they shouldn't. Explicit boundaries prevent this. Think of it as a "fence" around preserved code.
requesting-code-review Add a design-review prompt template alongside the code-review template. The design-review subagent reads plan docs, not diffs. Code review catches implementation bugs. Design review catches architectural gaps. Both are needed at different stages.
verification-before-completion Add a "regression" category to the verification table: "Existing tests still pass" with same iron-law enforcement. The current skill verifies new work. Extending it to verify preserved work means one skill covers both forward and backward verification.
test-driven-development Add a "negative test case" pattern: "given X, this should NOT happen." Add user story → test mapping format. TDD focuses on "what should work." Negative cases focus on "what should fail." Both are needed for complete coverage.

How to Build These Skills

You already have Superpowers' writing-skills skill, which teaches how to create new skills. Each new skill follows the same structure as existing Superpowers skills:

skills/ codebase-archaeology/ SKILL.md # Main skill doc (announcement, steps, deliverables, red flags) impact-analysis/ SKILL.md templates/ preservation-matrix.md # Template for the capability table invariants-checklist.md # Template for security invariant verification design-review/ SKILL.md design-reviewer-prompt.md # Subagent prompt (like code-quality-reviewer-prompt.md) task-briefing/ SKILL.md templates/ TASK-BRIEF-TEMPLATE.md # The standard brief format regression-verification/ SKILL.md templates/ regression-results.md # Template for capability-by-capability verification post-build-review.md # Template for design-vs-actual comparison
Build order recommendation: Start with codebase-archaeology (it feeds everything else) and impact-analysis (it's the biggest gap). Then task-briefing, design-review, and regression-verification. Each skill can be built and tested independently.
