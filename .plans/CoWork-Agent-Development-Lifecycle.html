Why Your Current Process Has Gaps

Your current 7 steps are solid for the design-to-build pipeline. But they assume a human developer who can discover context as they go — noticing existing patterns, asking clarifying questions mid-code, recognizing when a change might break something upstream. AI coding agents don't do that. They build what's described, exactly as described.

The core insight: Every gap we found in this session (SPIRE references, sidecar coupling, missing full pattern flow, peer review gaps) happened because the agent didn't have enough context BEFORE it started working. The missing steps are all about building that context.

Think of it this way: a human contractor can walk the job site, tap the walls, and say "there's plumbing behind this — I'll route around it." An AI coding agent gets the blueprints and starts cutting. The missing steps are the site survey — everything the agent needs to know before it writes a single line of code.

Context-First Development (CFD)

Build the map before you build the code. The human is the architect. The agent is the builder.

The name captures what makes this different from a standard dev process: context comes first, code comes second. Every step before coding exists to build the context document that an AI agent needs to build correctly. The output of the pre-coding phases isn't just "a plan" — it's a machine-readable specification that any coding agent can follow.

The Complete Process: 4 Phases, 13 Steps

The process is organized into 4 phases. Each phase has a clear owner (human, agent, or both) and produces specific deliverables that feed the next phase. The key insight: Phases 1-2 are human-led (building context), Phase 3 is agent-led (building code), and Phase 4 is both (verifying together).

1 DISCOVER — Map the Territory
Owner: Human (with agent assistance for exploration) · Goal: Know what exists before changing anything
NEW — MISSING FROM YOUR PROCESS
Step 1.1: Current State Inventory
Map every capability, endpoint, flow, and security property the system currently has. This is the "before" picture. Without it, you can't prove "after" doesn't break things.

What we learned: We had to do this mid-session when you said "you missed the full flow." If we'd done it first, every subsequent document would have been complete from the start.

Deliverable: capability-inventory.md — every endpoint, flow, data model, security invariant, and integration point.

Agent role: Agent explores codebase — the agent is excellent at this. Give it the repo and ask it to map everything. Human reviews and corrects.
HUMAN + AGENT
Step 1.2: Brainstorm / Pattern Input
Your existing Step 1. Human provides the idea, pattern, or requirement. This is the "what we want to build" input.

Enhancement: If there's a source pattern (like the Ephemeral Agent Credentialing Pattern v1.2), link it explicitly and have the agent extract its requirements as a checklist.

Deliverable: pattern-requirements.md — extracted "MUST" and "SHOULD" requirements from the pattern/idea.
NEW — MISSING FROM YOUR PROCESS
Step 1.3: Constraint & Compliance Check
Compare the current state inventory against the pattern requirements. Identify: what's already compliant, what's partially compliant, what's missing, and what's non-compliant. Also identify constraints — things you explicitly opted OUT of (like SPIRE) and why.

What we learned: The SPIRE confusion happened because we didn't document "we use Ed25519 challenge-response INSTEAD of SPIRE" upfront. This step prevents that.

Deliverable: compliance-matrix.md — requirement-by-requirement status with evidence from the codebase.
▼
2 DESIGN — Architect the Change
Owner: Human (agent drafts, human decides) · Goal: Design the change AND prove it doesn't break anything
EXISTING — ENHANCED
Step 2.1: PRD + Architecture Decisions
Your existing Steps 2-3 combined. Write the PRD and spec. But add a critical section: "What MUST NOT Change" — an explicit list of capabilities, invariants, and behaviors that the design must preserve.

Enhancement: Architecture Decision Records (ADRs) for every "do we need X?" question. Format: Context → Decision → Consequences. The sidecar question was an ADR that didn't have a formal home.

Deliverables: prd.md spec.md adrs/ (one file per decision)
NEW — MISSING FROM YOUR PROCESS
Step 2.2: Impact Analysis & Capability Preservation Proof
For every capability in the inventory (Step 1.1), explicitly state: PRESERVED (zero changes), ENHANCED (backward-compatible addition), NEW (didn't exist before), or REMOVED (intentionally dropped, with justification).

What we learned: This was the entire "Full Pattern Flow" section we added. You said "we can't fix one thing and break other things." This step is the formal version of that concern.

Rule: If you can't fill this table completely, you don't understand the change well enough to build it.

Deliverable: impact-analysis.md — capability-by-capability preservation matrix + security invariant verification.
EXISTING — ENHANCED
Step 2.3: Implementation Details
Your existing Step 4. Design the implementation. But add: code paths affected (which files/functions change) AND code paths NOT affected (which files/functions must remain untouched).

Why this matters for agents: Without the "do not touch" list, an agent might refactor a function it shouldn't, or add a parameter that breaks an interface contract.

Deliverable: implementation-details.md — with explicit "change" and "do not change" sections.
NEW — MISSING FROM YOUR PROCESS
Step 2.4: Peer Review Gate
Before breaking into tasks, get the design reviewed. This can be a human reviewer, a different AI agent asked to critique the design, or both.

What we learned: The 3rd-party developer review found 7 implementation gaps AFTER we'd already designed everything. If we'd had this gate before task breakdown, we wouldn't have had to revise the design.

Format: Reviewer identifies gaps → each gap gets a decision (with rationale) → decisions are documented in the design doc, not in chat.

Deliverable: review-decisions.md — gap identified → decision → rationale → phase assigned.
▼
3 BUILD — Agent Executes
Owner: Agent (human reviews) · Goal: Build exactly what was designed, nothing more, nothing less
NEW — MISSING FROM YOUR PROCESS
Step 3.0: Agent Context Briefing
Compile a single document that the agent reads before writing any code. This is NOT the PRD or spec — it's a briefing that tells the agent: here's what exists, here's what we're building, here's what you must not touch, and here's how to verify your work.

Why this is critical: An agent reading a PRD doesn't know the codebase. An agent reading the codebase doesn't know the plan. The briefing bridges both. Think of it as the "CLAUDE.md for this specific task."

Contents: Links to capability inventory, impact analysis, implementation details, "do not touch" list, test commands, and success criteria.

Deliverable: TASK-BRIEF.md — the one doc the agent reads first.
EXISTING
Step 3.1: Task Breakdown
Your existing Step 5. Break implementation into tasks. Enhancement: each task should include a "verify" step (not just "build X" but "build X and run Y to prove it works").

Deliverable: tasks.md — ordered task list with dependencies, each task has: description, files to change, files NOT to change, verification command.
EXISTING
Step 3.2: User Stories + Negative Cases
Your existing Step 6. Write user stories that can be tested. Enhancement: add negative test cases — "given X, this should NOT happen." Agents are good at building what should work; they're less good at anticipating what should fail.

Deliverable: user-stories.md — stories with acceptance criteria AND negative cases.
AGENT EXECUTES
Step 3.3: Implementation
The agent reads the briefing (Step 3.0), picks up tasks (Step 3.1), and builds. Human reviews PRs. Agent runs verification after each task.

Agent workflow: Read TASK-BRIEF.md → pick next task → implement → run task verification → commit → pick next task.
▼
4 VERIFY — Prove It Works, Prove Nothing Broke
Owner: Both · Goal: Verify the build matches the design AND existing capabilities survive
EXISTING — ENHANCED
Step 4.1: Verification Tasks
Your existing Step 7. Run the verification tasks from the user stories. Enhancement: run them in Docker (per your live test rules), not just locally.

Deliverable: test-results.md — terminal evidence of passing tests.
NEW — MISSING FROM YOUR PROCESS
Step 4.2: Regression Verification
Go back to the capability preservation matrix (Step 2.2). For every capability marked "PRESERVED," run a test proving it still works. For every capability marked "ENHANCED," run tests proving both old and new behavior work.

What we learned: "We can't fix one thing and break other things." This step is the formal proof. Run the existing test suite + targeted tests for preserved capabilities.

Deliverable: regression-results.md — capability-by-capability verification with evidence.
NEW — MISSING FROM YOUR PROCESS
Step 4.3: Post-Build Review
Compare what was built against the design. Did the implementation drift? Were any "do not touch" files modified? Do the actual code paths match the implementation details?

Agent role: Agent reviews diff — give the agent the full diff and the implementation details doc. Ask it: "does this diff match the design? What changed that wasn't planned?"

Deliverable: post-build-review.md — design vs. actual comparison, drift identified, decisions on drift.

Who Does What: Human vs. Agent

Human Leads (Phase 1-2)

The human is the architect. They provide the vision, make decisions, identify constraints, and approve the design. The agent assists with exploration (mapping the codebase, extracting pattern requirements) but the human makes every decision.

Key principle: The human builds context. The agent can help explore, but the human decides what matters.
Agent Leads (Phase 3)

The agent is the builder. It reads the briefing, picks up tasks, implements, and verifies. The human reviews PRs and makes judgment calls on edge cases the agent flags.

Key principle: The agent builds code. But only within the boundaries the human set in Phases 1-2. No improvisation.
Phase 4 is shared. The agent runs tests and generates verification evidence. The human reviews the evidence and makes the final "ship it" call. The agent can also review its own diff against the design doc — a form of self-audit.

Making It Work Across Coding Agents

Since you use multiple AI coding tools, the process docs should be agent-agnostic. The key is: every deliverable is a markdown file in the repo. Any agent that can read files can follow the process. Here's how it maps:

Agent How It Reads the Process Key File
Claude Code / Cowork CLAUDE.md references the process. TASK-BRIEF.md is read at session start via "Read TASK-BRIEF.md first" instruction. CLAUDE.md → TASK-BRIEF.md
Cursor / Windsurf .cursorrules or .windsurfrules references TASK-BRIEF.md. Agent reads it as project context. .cursorrules → TASK-BRIEF.md
Copilot Workspace TASK-BRIEF.md provided as specification input when starting a task. TASK-BRIEF.md
Any agent with file access TASK-BRIEF.md is a standard markdown file. Any agent that reads files can follow it. TASK-BRIEF.md
The TASK-BRIEF.md format is the universal interface. It doesn't matter which agent reads it — the content is the same. The only thing that changes is HOW each agent discovers it (CLAUDE.md for Claude, .cursorrules for Cursor, etc.).

TASK-BRIEF.md Template

This is the most important deliverable in the process. It's what the agent reads first. Here's the template:

# Task Brief: [Feature Name] ## Context - What this feature does (1-2 sentences) - Why we're building it (the problem it solves) - Link to PRD: ./prd.md - Link to pattern: [URL or file] ## Current State - Link to capability inventory: ./capability-inventory.md - Key capabilities that MUST be preserved: [list the critical ones] ## Design Decisions - Link to impact analysis: ./impact-analysis.md - Link to ADRs: ./adrs/ - Key constraints: [list non-obvious constraints] - OPTED OUT OF: [things we explicitly don't use, and why] ## Implementation Rules - Link to implementation details: ./implementation.md - DO NOT TOUCH: [files/functions that must not be modified] - CHANGE: [files/functions that should be modified] - NEW: [files/functions to create] ## Tasks - Link to tasks: ./tasks.md - Link to user stories: ./user-stories.md ## Verification - Run after each task: [command] - Run after all tasks: [command] - Regression test: [command or checklist] - Success criteria: [what "done" looks like]
