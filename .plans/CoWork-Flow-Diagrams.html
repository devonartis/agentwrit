AgentAuth Process & Flow Analysis

Date: 2026-03-01
Lens: Nine critical operational processes evaluated for design quality, completeness, and production readiness.
Scope: Operator workflows, developer experience, running application resilience.

Three Personas & Their Needs

Operator / Admin

Responsible for deploying, configuring, and managing AgentAuth infrastructure and onboarding apps.

What They Need:

  • API to register and manage apps
  • Dashboard to list, monitor, revoke apps
  • Audit trail attributed to app names, not opcodes
  • Secret rotation without full restarts
  • Metrics: token issuance, failures, latency per app
  • Safe secret distribution

3rd-Party Developer

Building an AI agent that needs scoped credentials from AgentAuth. Onboards through the sidecar.

What They Need:

  • Self-service registration flow
  • SDK or library for token management
  • Auto-renewal without custom loops
  • Clear documentation on integration
  • Visible broker health signal
  • Graceful failure modes

Running Application

An agentic app in production that depends on AgentAuth for credentials at runtime.

What They Need:

  • Token requests with low latency (<10ms)
  • Survive sidecar/broker restarts
  • Graceful degradation on network failures
  • Push revocation for security incidents
  • No concurrent activation bottlenecks
  • Transparent token renewal

Process 1: Operator Sets Up AgentAuth

PARTIALLY BROKEN

What the operator must do initially:

1. Generate and distribute AA_ADMIN_SECRET BROKEN Master key must be shared with every sidecar. No key rotation mechanism.
2. Deploy broker as single instance BROKEN No HA support. Single broker = single point of failure for all credential ops.
3. Initialize SQLite database OK Works, but no migration tool or backup guidance.
4. Document scope taxonomy for apps BAD DESIGN No API to define scopes. Manual process with no enforcement.
5. Enable broker API routes BAD DESIGN No feature flags. All admin endpoints live. No rate limiting by default.
6. Start broker and verify health BAD DESIGN Manual health checks. No startup diagnostics or integration tests included.
Verdict: Initial setup works, but secret distribution is unsafe and broker lacks HA. Operator has no UI/CLI to manage the system once running.

Process 2: Operator Onboards a New App

BLOCKER

What happens today when operator wants to add a new app:

1. Decide app scopes and name OK Manual process, no validation.
2. Edit docker-compose.yml or systemd config BROKEN Infrastructure change required. Should be API call.
3. Paste AA_ADMIN_SECRET into env BROKEN Master key copied again. Multiplies exposure risk.
4. Set AA_SIDECAR_SCOPE_CEILING BAD DESIGN Scope governance is per-sidecar, not per-app. Inflexible.
5. Deploy sidecar container/binary BROKEN Each app needs separate sidecar. Scaling issue.
6. Sidecar self-activates with admin secret BAD DESIGN No app entity created until sidecar first calls broker. Invisible to operator.
7. Broker receives activation, stores sidecar BROKEN No app record. Only sidecar ID persisted. No deregistration support.
Verdict: BLOCKER. This flow requires infrastructure changes for every new app. Admin secret is spread across all sidecars. There is no app-as-first-class-entity. No deregistration, no app-level audit, no app-level revocation possible. Operator has zero visibility until after deployment.

Process 3: Token Proxy Bootstraps

BAD DESIGN

Sidecar bootstrap sequence (on startup):

1. Sidecar health-checks broker OK Simple GET /v1/health call.
2. Sidecar authenticates with admin secret BROKEN Uses AA_ADMIN_SECRET. Should use ephemeral activation token instead.
3. Receives admin JWT (300s TTL) BROKEN Long-lived admin token in memory. Should be ephemeral.
4. Creates sidecar activation record BAD DESIGN Self-provisioning with admin token. No operator approval or audit.
5. Receives sidecar bearer token (900s TTL) OK Scoped to sidecar ID. Reasonable TTL.
6. Registers token routes and starts service OK Routes work correctly.
7. Starts renewal goroutine (80% TTL) OK Proactive renewal prevents token expiration.
8. Initializes circuit breaker for broker connection BAD DESIGN Circuit breaker stores cached tokens but they may be stale after broker restart.
Verdict: Bootstrap works, but relies on admin secret. Self-activation is a security anti-pattern. Circuit breaker caches create stale token issues. No operator visibility into this process.

Process 4: Developer Gets a Token (First Time)

WORKS (WITH FLAWS)

Developer app requests a token for a new agent:

1. App calls sidecar POST /v1/token {name, scope, task} OK Clear API, scoped request.
2. Sidecar validates scope vs. ceiling OK Proper authorization check.
3. Sidecar lazy-registers agent (5 broker round-trips) BAD DESIGN First request is slow: admin auth, launch token, challenge, sign, register. ~100ms+.
4. Sidecar uses admin secret for launch token BROKEN Runtime use of admin secret. Should pre-cache or use delegation.
5. Broker issues challenge (nonce, 30s TTL) OK Ed25519 challenge-response works correctly.
6. Sidecar generates Ed25519 keypair, signs nonce OK Client-side crypto correct.
7. Broker registers agent and returns internal token OK SPIFFE ID generation correct.
8. Sidecar stores agent in memory registry OK Subsequent requests fast.
9. Broker exchanges internal token for EdDSA JWT OK Token signed, scoped, 300s TTL.
10. Sidecar caches token for failsafe OK Circuit breaker protection.
11. Returns {access_token, expires_in, agent_id} to dev app OK Clear response format.
Verdict: Core protocol works well. Challenge-response auth is solid. But runtime admin secret usage is a gap. First-time latency penalty (100ms+) for 5 round-trips is high. No SDK support means developers hand-code all of this.

Process 5: Token Validation at Resource Server

BAD DESIGN

Resource server receives token and validates it:

1. Resource server receives Authorization: Bearer <token> OK Standard HTTP bearer token pattern.
2. Resource server has no JWKS endpoint BROKEN Cannot validate JWT locally. Must call broker every request.
3. Resource server calls broker POST /v1/token/validate BROKEN Network call per request. 5-10ms latency multiplied across all requests.
4. Broker validates signature and checks revocation BAD DESIGN Validation is correct but creates bottleneck. No scope middleware library.
5. Returns {valid: true, claims} OK Response format reasonable.
6. Resource server decides to accept or reject BAD DESIGN No middleware or library support. Developer must hand-code authorization.
Verdict: Validation logic is correct but architecture is backwards. Without JWKS, every request hits broker. For 100 req/sec, that's 100 calls/sec to broker. No support libraries mean developers implement authorization by hand.

Process 6: Operator Investigates Security Incident

BROKEN

Operator discovers suspicious activity and investigates:

1. Check audit log for compromised app BROKEN Audit attributed to sidecar_id, not app name. Cannot correlate to actual app.
2. Query by app name (e.g., "verify-agent") BROKEN App names not stored. No way to search audit by app.
3. Find all agents registered to that app BROKEN No app entity. Cannot list agents per app.
4. Revoke all tokens for that app BROKEN No app-level revocation. Must revoke by sidecar_id or individual agent.
5. Check if revocation propagated to sidecars BAD DESIGN No push notification. Sidecars must ask broker next validation. Delay of up to 5min.
6. Disable the app entirely BROKEN No app disable mechanism. Must stop/remove sidecar infrastructure.
Verdict: BROKEN. No app-level visibility. Cannot query audit by app. Cannot revoke by app. Revocation is pull-based (passive). Operator cannot quickly contain an incident affecting a single app.

Process 7: Operator Rotates Admin Master Key

BROKEN

Operator needs to rotate the AA_ADMIN_SECRET:

1. Generate new AA_ADMIN_SECRET OK Straightforward crypto.
2. Store new secret in broker config BROKEN No broker API to rotate. Must manually update config file.
3. Restart broker to accept new secret BROKEN Forces broker downtime. All tokens invalidated during restart.
4. Update AA_ADMIN_SECRET in all 10 sidecars BROKEN Must deploy to all sidecars simultaneously. Infrastructure change × 10.
5. Restart all sidecars with new secret BROKEN Sidecars restart, lose agent registry. Agents re-register (5 round-trips each).
6. Coordinate all restarts within window BROKEN Complex operational coordination. High risk of partial failures.
7. Verify all sidecars connected and healthy BAD DESIGN Manual verification. No automation or safety checks.
Verdict: BROKEN. Secret rotation requires broker + all sidecars to restart. Total service disruption. All agents must re-register. No support for rolling rotation. For 10 apps × 50 agents = 500 agents re-registering = massive latency spike.

Process 8: App Survives Broker Restart

BAD DESIGN

Broker is restarted (maintenance, patch, crash):

1. Broker process terminates OK Normal shutdown.
2. All EdDSA signing keys regenerated BROKEN Broker generates new keys on startup. All issued tokens now have invalid signatures.
3. SQLite loads audit and revocation data OK Persistent state restored correctly.
4. Sidecars detect broker is unhealthy, CB opens BAD DESIGN Circuit breaker serves cached tokens. Problem: cache has old keys.
5. Cached tokens fail validation at resource servers BROKEN Signature mismatch. All requests to resource servers fail.
6. Sidecars re-bootstrap with admin secret BAD DESIGN Sidecars wait for broker health, then re-auth.
7. All agents in sidecar registry re-register BROKEN Registry lost on sidecar restart. 5 round-trips per agent. 50 agents = 250 round-trips.
8. New tokens issued, requests resume BROKEN 5-10 minute recovery time. Latency spike during re-registration.
Verdict: Broker restart is a cascading failure. New keys invalidate all cached tokens. Sidecars experience outage until they re-bootstrap. All agents must re-register. For enterprise scale (10 apps, 50 agents each), this is 500 re-registrations = ~50 seconds latency spike.

Process 9: Do We Need the Token Proxy?

DESIGN QUESTION

Architectural question: Is the sidecar proxy the right pattern?

1. Sidecar adds latency to every token request BAD DESIGN Even cached requests route through sidecar. Network hop cost.
2. Sidecar is single per app, not per agent BAD DESIGN Bottleneck for multiple concurrent agents. SQLite BUSY errors under load.
3. Sidecar doesn't validate tokens locally BAD DESIGN Still calls broker for every validation. No caching of JWKS.
4. Sidecar restart = credential loss for app BROKEN Agent registry in memory. Lost on restart. Rebuilding is 5 round-trips per agent.
5. Alternative: Direct SDK approach BETTER App includes SDK, calls broker directly. No proxy latency. Validation via JWKS.
6. Alternative: HTTP proxy (optional) BETTER For non-SDK apps, optional lightweight proxy. Not mandatory for onboarding.
Verdict: Sidecar proxy pattern creates more problems than it solves. Mandatory proxy = bottleneck. Better architecture: SDK for direct broker access, optional proxy for legacy apps. Eliminates latency, complexity, and restart issues.

Proposed Architecture: Direct-to-Broker Model

Three integration paths instead of mandatory proxy:

Path 1: SDK (Recommended)
App embeds AgentAuth SDK → Direct HTTPS to Broker → Local JWKS validation → No proxy latency. Supports: auto-renewal, delegation chains, token caching, scope validation.
Path 2: Proxy (Optional, for Legacy Apps)
App without SDK → Optional lightweight proxy (can be sidecar or separate) → Acts as cache/renewal layer → Reduces latency. Not mandatory for onboarding.
Path 3: HTTP Direct (Simple Cases)
Stateless app → Direct HTTP to Broker (no TLS certs initially) → Works for internal networks → Simplest onboarding, lowest latency.

Benefits: Eliminates proxy bottleneck. Reduces latency. Solves broker restart cascade (app can validate tokens locally). Improves scalability (no sidecar BUSY errors). Faster onboarding (no sidecar container required).
