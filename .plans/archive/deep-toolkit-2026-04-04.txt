The Complete AI Agent Toolkit
=============================================================
Agents, Sub-Agents, Prompts, Hooks, and Skills derived from
208+ sessions, 240 facet analyses, 4 insight reports, and
2,970 messages of real production use.

Everything here traces to a specific failure or success pattern.
Nothing is hypothetical.


========================================================================
SECTION 1: AGENTS AND SUB-AGENTS
========================================================================
These are autonomous or semi-autonomous agent patterns that solve
structural problems no single prompt can fix.


AGENT 1: THE RESEARCHER (read-only scout)
-------------------------------------------
Problem it solves: Agent starts coding before understanding the codebase.
Entire feature branches got deleted because the agent built on assumptions
instead of reading existing code. (257 wrong-approach events)

How it works:
- Spawned as a sub-agent with READ-ONLY permissions (Read, Grep, Glob,
  WebFetch only -- no Edit, Write, or Bash)
- Given a specific question: "How does auth work in this codebase?"
- Returns a structured findings report with file paths and line numbers
- The orchestrating agent cannot proceed until the researcher reports back

Why a sub-agent and not a prompt:
A prompt says "read first." The agent nods and starts coding anyway.
A read-only sub-agent physically cannot edit files. The constraint is
structural, not behavioral.

Generic implementation:
  Spawn agent with tools: [Read, Grep, Glob, WebSearch]
  Prompt: "You are in research mode. You CANNOT edit files.
  Research: [question]. Return: what exists today, how it works,
  what conventions it follows. Cite file:line for every claim."


AGENT 2: THE COUNCIL (multi-perspective review)
-------------------------------------------------
Problem it solves: The author of a plan/spec/test has blind spots.
A single reviewer shares the same context and misses the same things.

How it works:
- 3 sub-agents review the same artifact from different perspectives
- Agent A: "Review for technical correctness and implementation feasibility"
- Agent B: "Review for security gaps and missing edge cases"
- Agent C: "Review for user experience and real-world usability"
- Each returns independent findings
- Orchestrator synthesizes and presents conflicts

CRITICAL LESSON from real sessions:
The council's quality depends entirely on how well you brief the
sub-agents. Giving them "review this spec" produces generic feedback.
Giving them the project context, the specific concerns, and what to
focus on produces findings that catch real blind spots.

Bad briefing: "Review this spec for issues."
Good briefing: "This spec defines token renewal for a security system.
The system uses Ed25519 JWT tokens with 5-minute default TTL. Review
for: (1) any scenario where a revoked token could still be renewed,
(2) any TTL manipulation that bypasses the MaxTTL ceiling, (3) any
state where the old token isn't revoked after renewal succeeds.
The spec is at [path]. The relevant code is in [package]."


AGENT 3: THE CONTAMINATION SCANNER
------------------------------------
Problem it solves: When forking, extracting, or cleaning up a codebase,
unwanted code from the parent project leaks through. Happened during
sidecar removal (B0), HITL removal (Python SDK), and doc cleanup.
Code review caught contamination in docs/ that the plan missed.

How it works:
- Given a list of forbidden terms/patterns (enterprise feature names,
  deprecated imports, internal references)
- Scans ALL files, not just source code (docs, configs, scripts, tests,
  README, CHANGELOG, comments)
- Reports every match with file, line number, and surrounding context
- Runs as a gate: blocks merge/PR until clean

Why this needs to be an agent, not just grep:
Grep finds literal matches. The agent finds semantic contamination --
a doc that describes HITL workflow without using the word "HITL," or
an example that demonstrates an enterprise-only flow using generic
variable names. The agent understands intent, not just patterns.

Generic implementation:
  Spawn agent with tools: [Read, Grep, Glob, Bash]
  Prompt: "Scan the entire repo for contamination from [source project].
  Forbidden terms: [list]. Check ALL file types including docs, configs,
  scripts, tests, and markdown. Report every match. Also flag any code
  that DESCRIBES forbidden functionality without using the exact terms.
  A clean report means zero matches. Anything else blocks the merge."


AGENT 4: THE EVIDENCE RECORDER
---------------------------------
Problem it solves: Test evidence comes back as AI-generated summaries,
rendered markdown, or pre-written "PASS" verdicts. Real evidence must
be verifiable terminal output that someone who wasn't there can audit.

How it works:
- Runs each test command and captures ACTUAL output
- Writes evidence files with mandatory structure:
  WHO (persona), WHAT (plain English), WHY (what breaks if this fails),
  COMMAND (exact, copy-pasteable), OUTPUT (verbatim), VERDICT (earned)
- The verdict is written AFTER seeing output, never before
- If a test fails, it records the failure. No silent retries.

Key anti-pattern it prevents:
"Self-declared passes" -- the most trust-damaging behavior observed.
The agent writes PASS before running the test, or summarizes output
instead of including it verbatim. This corrupts the evidence trail
and means nobody can verify what actually happened.

Generic implementation:
  Spawn agent with tools: [Bash, Write]
  Prompt: "You are recording test evidence. For each test:
  1. Write the banner (WHO/WHAT/WHY) BEFORE running the command
  2. Run the EXACT command and capture ALL output
  3. Include output VERBATIM in a code block. Never summarize.
  4. Write VERDICT only after seeing the output
  5. If it fails, record FAIL. Do not retry without documenting."


AGENT 5: THE DRIFT DETECTOR (session watchdog)
------------------------------------------------
Problem it solves: Agent drifts from the plan on step 3+ and builds
an entire structure on a wrong foundation before anyone notices.
Only 20% of sessions fully achieved their goals. 40% were "mostly"
achieved -- meaning drift happened but wasn't caught in time.

How it works:
- Runs as a background agent alongside the main coding agent
- After every N tool calls or every file edit, it checks:
  "Is the current action consistent with the stated plan?"
- If drift is detected, it interrupts with a checkpoint
- Reports: what the plan says, what the agent is actually doing,
  and where they diverge

Why this matters:
Satisfaction is bimodal. Sessions either go well or they don't.
There's very little "medium." The ROI of catching drift early is
enormous because a bad start rarely recovers incrementally.


AGENT 6: THE VOICE MATCHER
-----------------------------
Problem it solves: When writing user-facing content (README, docs,
blog posts, project notes), the agent defaults to generic "AI voice"
instead of matching the user's established tone. Verbatim logging
requests get rewritten. Casual notes become formal documentation.
Sales-pitchy copy appears in technical READMEs. (Distinct from
factual accuracy -- this is about tone.)

How it works:
- Given 3-5 examples of the user's existing writing from the project
- Analyzes voice characteristics: formality level, sentence length,
  use of jargon, first/third person, active/passive voice
- Produces a voice profile that the writing agent must follow
- Reviews generated content against the profile before presenting

Generic implementation:
  Step 1: Read 3-5 existing docs from the project
  Step 2: Produce voice profile (tone, formality, sentence structure,
          vocabulary patterns, what to avoid)
  Step 3: Pass profile as a constraint to the writing agent
  Step 4: Review output against profile before presenting


AGENT 7: THE MULTI-REPO NAVIGATOR
------------------------------------
Problem it solves: When working across multiple project directories in
one session, the agent loses track of which repo it's in. Files get
edited in the wrong repo. Git commits land in the wrong project.
Commands run against the wrong codebase.

How it works:
- Maintains explicit state: "Currently in [repo] at [path]"
- Before every file edit or bash command, verifies the target path
  matches the intended repo
- When the user switches context ("now let's work on the SDK"),
  the agent explicitly acknowledges the switch and confirms the
  new working directory
- Refuses to edit files outside the declared working repo without
  explicit confirmation

Key finding: This is a structural issue. The bash shell resets cwd
between calls. The agent sometimes uses `cd` anyway, then subsequent
commands fail because cwd has reset. When this happens, the agent
often blames the environment instead of recognizing its own mistake.

PROMPT 8: THE ANTI-HALLUCINATION CHECK
-----------------------------------------
Problem it solves: Agent includes fake install instructions, references
non-existent functions, or describes features that don't exist. Found
in documentation sessions where Claude invented plausible-looking but
false installation commands for a package not on PyPI.

---
Before presenting this [document/README/guide] to me, verify:

1. Every command you included -- does it actually work? If you wrote
   "pip install X", is X actually on PyPI? If you wrote "go get X",
   does that module exist?
2. Every function/type/API you referenced -- grep the codebase to
   confirm it exists. If it doesn't, remove the reference.
3. Every URL you included -- is it real or did you guess it?

Flag anything you're uncertain about with [UNVERIFIED] rather than
presenting it as fact. I'd rather see honest gaps than confident lies.
---


PROMPT 9: THE RECOVERY PROMPT (when things go sideways)
---------------------------------------------------------
Problem it solves: When the agent's understanding gets too corrupted
to correct incrementally, you need a full reset. Discovered in sessions
where 4+ corrections didn't fix the agent's mental model and the user
had to create a "triage document" to start over.

---
Stop. We're resetting.

Your current understanding of [topic] is wrong. Here's what's
actually true:

1. [Fact 1]
2. [Fact 2]
3. [Fact 3]

Do NOT reference anything you said before this message. Treat this
as your new starting point. Restate these facts back to me, then
tell me how they change your approach.
---


PROMPT 10: THE SUB-AGENT BRIEFING TEMPLATE
---------------------------------------------
Problem it solves: Council/review sub-agents produce generic feedback
because the orchestrating agent gives them inadequate context.
The quality of sub-agent output is entirely determined by the quality
of the briefing. (Found in council review sessions where false
positives wasted time.)

---
You are reviewing [artifact type] for [specific concern].

Context:
- This system does [1-2 sentence description]
- The key constraint is [what matters most]
- The specific risk is [what could go wrong]

The artifact is at [path].
The relevant source code is in [package/directory].

Focus your review on:
1. [Specific question 1]
2. [Specific question 2]
3. [Specific question 3]

Do NOT give generic feedback. Every finding must reference a specific
line or section and explain the concrete impact.
---


PROMPT 11: THE "DON'T APOLOGIZE, ADAPT" PROMPT
-------------------------------------------------
Problem it solves: The "hollow apology loop" -- agent gets corrected,
says "you're right, my bad," then makes the same mistake. The apology
becomes a substitute for behavioral change. Named as a recurring
anti-pattern across multiple sessions.

---
When I correct you, do not apologize. Instead:

1. State what you'll do differently (one sentence)
2. Do it

"Sorry" tells me nothing. Changed behavior tells me everything.
If you catch yourself about to write "you're right" or "my apologies,"
skip it and go straight to the fix.
---


PROMPT 12: THE SCOPE LOCK
----------------------------
Problem it solves: Agent expands scope. Asked for a high-level plan,
it writes a detailed spec. Asked for one function, it refactors the
file. Asked for phased work, it delivers a monolith. This is the
#1 friction source (257 events) manifesting in architecture work.

---
SCOPE LOCK: This task is exactly [X]. Nothing more.

Do not:
- Add related improvements you noticed
- Refactor surrounding code
- Expand the deliverable beyond what I described
- Add phases or steps I didn't ask for
- Create helper functions/files "while we're here"

If you see something that SHOULD be fixed but is outside this scope,
note it at the end as "Out of scope observation:" -- do not fix it.
---

HOOK 6: WORKING DIRECTORY GUARD
----------------------------------
Problem it solves: Agent uses `cd` in bash commands, then subsequent
commands fail because cwd resets between calls. Agent then blames
the environment. (Multiple sessions, 157 "Command Failed" events)

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "bash -c 'if echo \"$TOOL_INPUT\" | grep -qE \"^cd\\s|;\\s*cd\\s|&&\\s*cd\\s\"; then echo \"WARNING: Using cd in Bash tool. Shell state does not persist between calls. Use absolute paths instead.\" && exit 1; fi'"
      }
    ]
  }
}


HOOK 7: CROSS-REPO EDIT GUARD
---------------------------------
Problem it solves: Agent edits files in the wrong repo during
multi-project sessions. (Found: edits landing in code repo when
user wanted changes in Obsidian vault, commits in wrong repo)

// Set PROJECT_ROOT to your primary working directory
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit",
        "command": "bash -c 'EXPECTED_ROOT=\"/path/to/your/project\"; FILE=$(echo \"$TOOL_INPUT\" | head -1); if ! echo \"$FILE\" | grep -q \"$EXPECTED_ROOT\"; then echo \"BLOCKED: Editing file outside project root ($EXPECTED_ROOT). File: $FILE. Is this intentional?\" && exit 1; fi'"
      }
    ]
  }
}


HOOK 8: PRE-VERDICT EVIDENCE CHECK
-------------------------------------
Problem it solves: Agent writes PASS/FAIL verdicts in test evidence
before actually running the test. (Most trust-damaging anti-pattern
observed, specifically called out in acceptance testing sessions)

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "command": "bash -c 'if echo \"$TOOL_INPUT\" | grep -qiE \"verdict.*pass|verdict.*fail|PASS|FAIL\" && echo \"$TOOL_INPUT\" | grep -qiE \"evidence|test.*result|acceptance\"; then echo \"WARNING: Writing a test verdict. Confirm: was the test command actually executed and output captured BEFORE this verdict?\" && exit 1; fi'"
      }
    ]
  }
}


HOOK 9: SCOPE CREEP DETECTOR
-------------------------------
Problem it solves: Agent creates files that weren't in the plan.
New files appearing unexpectedly is a leading indicator of scope creep.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "command": "bash -c 'echo \"NOTE: Creating new file. If this file was not part of the agreed plan, confirm with user before proceeding.\"'"
      }
    ]
  }
}


HOOK 10: STALE CLAIM DETECTOR
--------------------------------
Problem it solves: Agent claims "fully oriented" or "I understand the
codebase" without having actually read the relevant files. (Multiple
sessions where premature orientation claims led to wrong work)

// This is a soft hook -- it logs, doesn't block
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "command": "bash -c 'if echo \"$TOOL_OUTPUT\" | grep -qiE \"fully oriented|I understand|I.ve read|I.m familiar\"; then echo \"REMINDER: Claims of understanding should be backed by specific file:line references. What exactly did you read?\"; fi'"
      }
    ]
  }
}

SKILL 8: /reset
-----------------
Problem it solves: Agent's mental model is too corrupted for
incremental correction. You need a clean restart without losing
session history. Discovered when users had to write "triage documents"
to fully reset the agent after 4+ corrections didn't stick.

---
# /reset

## Instructions

The user is resetting your understanding. Everything you believed
about the current task before this point may be wrong.

1. STOP all current work
2. Read the user's correction that follows this command
3. Restate their correction in your own words
4. Explain what you were doing wrong and how the correction changes
   your approach
5. Wait for confirmation before resuming any work

Do not reference your previous approach. Do not blend old assumptions
with new information. Start fresh from the user's correction.
---


SKILL 9: /brief-agents
------------------------
Problem it solves: Sub-agent quality depends entirely on briefing
quality. When the orchestrator gives vague prompts to council
reviewers or parallel workers, the results are generic or wrong.
(Found: council false positives from inadequate context)

---
# /brief-agents

## Instructions

Before spawning any sub-agent, write the briefing and show it to the
user for approval. Each briefing must include:

1. ROLE: What perspective this agent reviews from
2. CONTEXT: 3-5 sentences about the system and what matters
3. ARTIFACT: Exact path to what they're reviewing
4. SOURCE: Where to find the relevant code (package, directory)
5. FOCUS: 3 specific questions, not generic "review for issues"
6. FORMAT: How to structure findings (specific line refs required)

Do not spawn the agent until the user approves the briefing.
Vague briefings produce vague results. Specific briefings catch
real problems.
---


SKILL 10: /migrate
--------------------
Problem it solves: Code migration between repos (cherry-picks, forks,
extractions) has a repeatable process that gets violated when done
ad-hoc. (6 successful migration batches all followed the same steps;
failures happened when steps were skipped)

---
# /migrate

## Instructions

You are migrating code. Follow this exact sequence:

1. ANALYZE: Read the source commits. List every file changed,
   every new dependency, every behavioral change.
2. PICK: Apply changes. Resolve conflicts by keeping the TARGET
   repo's patterns, not the source's.
3. COMMENT: Add/update code comments explaining who calls what
   and why. Migration without comments compounds wrong assumptions.
4. TEST: Run gate checks (compile, unit tests, contamination scan).
   Fix failures before proceeding.
5. REGRESSION: If behavior changed, write regression tests BEFORE
   running the full suite so they're included.
6. VERIFY: Run acceptance tests against the live system.
   Record evidence per /evidence format.
7. DOCUMENT: Update CHANGELOG, decision log, and any affected docs
   in the SAME commit as the code change.

Do not skip steps. Do not combine steps. Each step must pass before
the next begins.
---


SKILL 11: /voice-check
------------------------
Problem it solves: Agent defaults to generic AI voice when writing
user-facing content. README copy becomes sales-pitchy. Project notes
become formal documentation. Verbatim requests get rewritten.

---
# /voice-check

## Instructions

Before writing any user-facing content (README, docs, blog posts,
project notes, commit messages), do this:

1. Read 3 existing documents from this project
2. Note: formality level, sentence length, vocabulary, first vs
   third person, active vs passive voice, use of jargon
3. Match that voice in your output

Rules:
- If the project writes casually, you write casually
- If the project uses technical jargon, you use the same jargon
- If the user asked for verbatim logging, do NOT rewrite their words
- Never add marketing language unless the existing docs have it
- When in doubt, match the most recent doc, not the oldest
---


SKILL 12: /plan-check
-----------------------
Problem it solves: "Planning in circles" -- agent produces meta-plans
about how it will plan, or iterates on the planning approach, rather
than producing the actual deliverable. User had to interrupt twice
in one session to stop the agent from re-planning.

---
# /plan-check

## Instructions

You are about to write a plan. Rules:

1. The plan is the DELIVERABLE, not a preamble to work
2. Write the actual plan. Not a plan to write a plan.
3. Each step must be concrete: what file, what change, what test
4. If you catch yourself writing "first we'll need to analyze..."
   that's meta-planning. Skip it. Just write the steps.
5. Maximum 1 paragraph of context before the numbered steps
6. If the plan exceeds 15 steps, split into phases with
   explicit checkpoints

Present the plan and STOP. Do not begin executing it.
---


SKILL 13: /multi-repo
-----------------------
Problem it solves: Working across multiple repos in one session causes
file edits in the wrong repo, git commits in the wrong project, and
commands running against the wrong codebase.

---
# /multi-repo

## Instructions

You are working across multiple repositories. Maintain explicit state.

1. At all times, declare which repo you're working in:
   "ACTIVE REPO: [name] at [path]"
2. Before every file edit, verify the file path starts with the
   active repo's root directory
3. Before every bash command, use absolute paths. Never use `cd`.
4. When switching repos, announce it:
   "SWITCHING TO: [name] at [path]"
5. If you need to reference a file in another repo without switching,
   say so: "Reading from [other repo] for reference only"

Never edit files outside the active repo without explicit user
confirmation.
---


SKILL 14: /debrief
--------------------
Problem it solves: Failures get fixed but not learned from. The same
category of error recurs across sessions because nobody extracted the
pattern. The most productive users treat every failure as a harness
engineering opportunity. (This user created process improvements after
failures, which is why their success rate held steady despite taking
on harder tasks over time.)

---
# /debrief

## Instructions

A failure just occurred. Before moving on:

1. WHAT happened: State the failure in one sentence
2. WHY it happened: Root cause, not symptoms
3. CATEGORY: Is this a new failure mode or a repeat?
   (wrong approach / misunderstood request / tool misuse /
   scope creep / drift / contamination / other)
4. PREVENTION: What prompt, hook, skill, or process change
   would prevent this category of failure in the future?
5. ACTION: Implement the prevention now, or log it as a task

Do not skip step 4. Fixing the symptom without addressing the
category means it will happen again.

========================================================================
SECTION 5: THE PATTERNS NOBODY TALKS ABOUT
========================================================================
Jewels from the data that don't fit neatly into a tool but change
how you think about working with AI agents.


PATTERN: FRICTION IS BIMODAL
Sessions either go well or they go badly. There's almost no middle
ground. "Happy" satisfaction appeared exactly 8 times across 200+
sessions. That number never moved. This means: preventing a bad
session start has 10x the ROI of incrementally improving a mediocre
one. Invest in session-start rituals, not mid-session fixes.


PATTERN: WRONG APPROACH IS INCREASING, NOT DECREASING
Across 4 reporting periods, wrong-approach events went from 188 to
273. Misunderstood requests went from 164 to 222. But fully-achieved
sessions stayed flat at ~19%. Translation: the user is taking on harder
tasks over time, and the agent's baseline reliability isn't keeping
pace. The toolkit (hooks, skills, prompts) is what holds the line.
Without it, the fully-achieved rate would be dropping.


PATTERN: ZERO-FRICTION SESSIONS HAVE A SIGNATURE
They're single-purpose. They use a skill as an entry point. There's
no ambiguity about output format. They're short. The insight: friction
correlates with ambiguity, not complexity. A complex task with clear
constraints succeeds. A simple task with unclear intent fails.


PATTERN: TEMPLATE-BASED CREATION BEATS OPEN-ENDED CREATION
When the user pointed to an existing exemplar and said "do this pattern
for [new thing]," the result was dramatically better than "create
[new thing] from scratch." Pointing at an existing file and saying
"follow this structure" gives the agent a concrete target instead of
an infinite design space.


PATTERN: THE 10/80 RULE FOR ESSENTIAL SESSIONS
Sessions rated "essential" all share: user spent 10-20% of the session
establishing constraints and correcting 1-2 early mistakes, then 80%
watching productive execution. The early investment pays off. Sessions
where the user didn't invest that upfront 10% were the ones that went
sideways.


PATTERN: EVENING SESSIONS DRIFT MORE
Peak message volume is 6pm-midnight, with significant work past
midnight. Late sessions show more drift, likely because the user is
tired and less likely to catch errors early. If you work late with AI
agents, use more aggressive checkpointing (every 2-3 changes instead
of every 5).


PATTERN: USER REJECTION RATE IS YOUR LEADING METRIC
"User Rejected" tool events (126-172 per period) are the canary in
the coal mine. A declining rejection rate means your hooks and rules
are working. A rising one means the agent is increasingly doing things
you don't want. Track this number. It tells you more than any
satisfaction survey.

SECTION 6: HOW EVERYTHING FITS TOGETHER
========================================================================

SESSION LIFECYCLE:

  /session-start ──> /research ──> /plan-check ──> [work begins]
       |                |              |                |
  Load memory      Read-only       No meta-         Phase-gated
  Confirm state    sub-agent       planning         execution
       |                |              |                |
       └── HOOKS: branch guard, push guard, cd guard ──┘
                        |
                /checkpoint (every 3-5 changes)
                        |
              [drift detector agent running]
                        |
                /simplify (strip over-engineering)
                        |
       /guard (contamination) + /evidence (test proof)
                        |
              /voice-check (if writing docs)
                        |
        /debrief (if anything went wrong)
                        |
              /session-end (capture lessons)


AGENT TOPOLOGY:

  ORCHESTRATOR (main agent, user-facing)
       |
       |── RESEARCHER (read-only, spawned before coding)
       |── COUNCIL (3 reviewers, spawned for spec/test review)
       |     |── Technical reviewer
       |     |── Security reviewer
       |     |── UX/usability reviewer
       |── CONTAMINATION SCANNER (spawned before merge)
       |── EVIDENCE RECORDER (spawned during testing)
       |── DRIFT DETECTOR (background, monitors orchestrator)
       └── VOICE MATCHER (spawned for user-facing content)


The orchestrator does the thinking. The sub-agents do the checking.
The hooks enforce the walls. The skills define the rituals.
The prompts shape the conversation.

None of this replaces human judgment. It channels agent energy so
that human judgment is spent on decisions that matter, not on
correcting mistakes that should have been prevented.
