Claude Code Insights - What I Learned From 208 Sessions
========================================================

1) What I'm doing well
-----------------------
- Built a real development process and actually enforce it. Gate testing
  between migration batches, acceptance criteria before merges, evidence
  files with banners. Most people skip this stuff.
- Knowledge management system is legit. Daily notes, weekly retros,
  structured memory files, custom skills for recurring workflows. Treating
  institutional memory as a first-class artifact, not an afterthought.
- Catch wrong approaches fast. When the agent goes off-rails, redirect
  immediately instead of letting bad work compound. That instinct saved
  entire sessions worth of wasted effort.
- Documentation-first mindset. Specs and plans before code. Rare, and it
  pays off in traceable, auditable work.

2) Where I could do better
---------------------------
- Sessions try to cover too much ground. Success rate drops when scope
  creeps. Smaller checkpoints with explicit "stop and show me" gates would
  convert "mostly done" into "fully done."
- Process rules are elaborate but buried. The agent keeps violating them
  because it doesn't internalize them deeply enough at session start. The
  rules need to be louder, not just present.
- Sometimes let the agent run too long before correcting. One time an
  entire feature branch got deleted because the wrong approach wasn't
  caught until it was fully built. Earlier "what's your plan?" checkpoints
  would catch drift sooner.
- Front-load constraints in prompts. Instead of correcting after the fact,
  say what NOT to do upfront: "use the SDK, not raw HTTP," "high-level
  only, don't write a spec."

3) Claude Code features to use better
--------------------------------------
- Hooks. Doing manual enforcement of things that should be automated. A
  pre-commit hook that checks branch names, validates test formats, or
  blocks pushes to main would eliminate entire categories of mistakes
  currently caught by eye.
- Headless mode. 63 documentation sessions with repetitive correction
  cycles. Batch doc generation with strict tool permissions would let you
  script the boring parts and only intervene on interesting decisions.
- Phase-gated prompting. Explicitly tell the agent to stop between phases.
  "Do Phase 1, then STOP. Do not proceed until I say continue." Matches
  how I already think but I'm not always enforcing it in the prompt.
- Parallel sub-agents for independent work. Migration batches had steps
  that could run simultaneously (tests, doc updates, contamination checks).
  Parallel agents would cut wall-clock time.

4) One golden nugget
---------------------
Code comments are the interface between human intent and agent execution.
Multiple agent sessions wrote and reviewed the same code without ever
flagging that the role model was undocumented. Each agent read the code,
made assumptions, and moved on. When comments were finally added explaining
who calls what and why, the agents stopped making wrong assumptions.
Strong comments don't just help humans. They're the difference between an
AI agent that compounds your intent and one that compounds its own mistakes.
