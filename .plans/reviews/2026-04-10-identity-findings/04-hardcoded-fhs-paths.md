# Finding 04: Hardcoded FHS Config Search Paths

**ID:** TD-CFG-002
**Severity:** HIGH (blocks rebrand)
**Status:** UNRESOLVED
**Audit reference:** C1-4

---

## What Was Found

In `internal/cfg/configfile.go`, three locations hardcode the brand name into filesystem paths:

### Line 21 — System config path

```go
locs = append(locs, "/etc/agentauth/config")
```

### Line 24 — Home directory config path

```go
locs = append(locs, filepath.Join(home, ".agentauth", "config"))
```

### Line 108 — Generated config file header

```go
content := fmt.Sprintf("# AgentAuth Configuration\n# Generated by: aactl init --mode=%s\n...", ...)
```

**Note:** The audit only caught line 21. Lines 24 and 108 were missed.

## Why This Is a Problem

1. **FHS violation under rebrand.** An operator rebranding to "AgentWrit" still has to put config files under `/etc/agentauth/` or the broker won't find them. The filesystem hierarchy forces the old brand.
2. **Home directory dotfile forces old brand.** `~/.agentauth/config` — same problem.
3. **Generated config file announces old brand.** The header comment says "AgentAuth Configuration" even if the product is now AgentWrit.

## Solution

### Option A: Make the entire search path configurable via `AA_ISSUER` / a brand variable

Derive the FHS path from a configured brand name. But this introduces a coupling between the issuer identity and the filesystem layout, which doesn't make sense — they're separate concerns.

### Option B: Remove hardcoded search paths, rely on explicit paths only (RECOMMENDED)

Remove the hardcoded `/etc/agentauth/config` and `~/.agentauth/config` entries. Config files are found via:

1. `AA_CONFIG_PATH` env var (already supported — highest priority)
2. `./config.yaml` in current working directory (sane default for local dev)
3. `--config <path>` CLI flag (to be added to broker startup)

This matches how most modern CLIs work: explicit flag > env var > cwd-relative fallback. No hardcoded FHS paths in source code. Operators who want FHS compliance set `--config /etc/agentwrit/config` in their systemd unit or `AA_CONFIG_PATH=/etc/agentwrit/config` in their env.

```go
func configLocations() []string {
    var locs []string
    // 1. Explicit env var (highest priority)
    if p := os.Getenv("AA_CONFIG_PATH"); p != "" {
        locs = append(locs, p)
    }
    // 2. CWD-relative fallback
    locs = append(locs, "config.yaml")
    // No hardcoded FHS paths — operator sets --config or AA_CONFIG_PATH
    return locs
}
```

### Fix the generated config header

```go
content := fmt.Sprintf("# AgentWrit Configuration\n# Generated by: awrit init --mode=%s\n...", ...)
```

Or better, derive the product name from a constant:

```go
const ProductName = "AgentWrit"
const CLIBinaryName = "awrit"
```

And use `ProductName` in the generated header. This constant is the ONE place the brand lives — everything else references it.

| File | Change |
|------|--------|
| `internal/cfg/configfile.go` | Remove `/etc/agentauth/config` and `~/.agentauth/config`; add `config.yaml` cwd fallback; fix generated header |
| `internal/cfg/configfile_test.go` | Update tests for new search path behavior |
| New: `internal/cfg/brand.go` or `internal/cfg/constants.go` | Define `ProductName` and `CLIBinaryName` constants |

## Migration Note

**BREAKING CHANGE for operators using the default config file search.** Operators who relied on the broker finding config at `/etc/agentauth/config` must either:

- Set `AA_CONFIG_PATH=/etc/agentauth/config` (or their new path)
- Use `--config /path/to/config` when starting the broker
- Place `config.yaml` in the working directory

This is a minor migration burden — the `AA_CONFIG_PATH` env var already exists and takes highest priority.
